## C++内存管理

### 1. 基本概念

- ***c++为什么没有垃圾回收机制来管理内存？***

  ​	C++的设计者Bjarne Stroustrup在我所知的唯一一本介绍语言设计的思想和哲学的著作《The Design and Evolution of C++》(中译本：C++语言的设计和演化)中花了一个小节讨论这个特性。简而言之，Bjarne本人认为，

  > ​	“我有意这样设计C++，使它不依赖于自动垃圾回收(通常就直接说垃圾回收)。这是基于自己对垃圾回收系统的经验，我很害怕那种严重的空间和时间开销，也害怕由于实现和移植垃圾回收系统而带来的复杂性。还有，垃圾回收将使C++不适合做许多底层的工作，而这却正是它的一个设计目标。但我喜欢垃圾回收的思想，它是一种机制，能够简化设计、排除掉许多产生错误的根源。

  - 统一的自动垃圾回收系统无法适用于各种不同的应用环境，而又不至于导致实现上的负担；
  - 没有应用可能因为有了垃圾回收而做得更好；

- ***c++目前的内存管理方法，或者说释放资源的方式有哪些？***
  - 手动释放
    - 即调用delete函数，现在c++工程几乎不这么写了
    - 多见于示例代码
  - RAII释放
    - 当所有拥有资源对象的生命周期结束以后，资源就会自动释放。
    - 智能指针就是典型地用RAII管理内存资源的做法
  - 内存池
    - **预先申请分配**一定数量、大小预设的内存块留作备用。当有新的内存需求时，就从内存池中分出一部分内存块，若内存块不够再继续申请新的内存，当内存释放后就回归到内存块留作后续的复用，使得内存使用效率得到提升，一般也不会产生不可控制的内存碎片。
  - GC垃圾回收机制
    - 虽然c++标准不带GC，但是还是有很多库实现了GC

### 2. 进阶概念

#### 2.1 RAII

- ***RAII实现原理？***

  ​	c++的手动申请的资源是不会自动释放的。所以我们希望通过一个方式去将这个资源自动释放，怎么做呢？

  ​	可以参考栈区中，局部变量在超出其生命域自动释放的做法。也就是说我们把手动申请的资源以局部变量地方式去释放。那做法就很直接了，只需要定义一个类，这个类是一个局部变量，在超出其生命域就会自动释放。然后用类将申请的资源封装起来即可。当超出其生命域的时候，析构函数就会被自动调用，完成资源释放。

#### 2.2 内存池

- ***经典内存池实现方式***

  - > [引用自](https://blog.csdn.net/K346K346/article/details/49538975)

  1. 先申请一块连续的内存空间，该段内存空间能够容纳一定数量的对象；
  2. 每个对象连同一个指向下一个对象的指针一起构成一个内存节点（Memory Node）。各个空闲的内存节点通过指针形成一个链表，链表的每一个内存节点都是一块可供分配的内存空间；
  3. 某个内存节点一旦分配出去，从空闲内存节点链表中去除；
  4. 一旦释放了某个内存节点的空间，又将该节点重新加入空闲内存节点链表；
  5. 如果一个内存块的所有内存节点分配完毕，若程序继续申请新的对象空间，则会再次申请一个内存块来容纳新的对象。新申请的内存块会加入内存块链表中。

  如上图所示，申请的内存块存放三个可供分配的空闲节点。空闲节点由空闲节点链表管理，如果分配出去，将其从空闲节点链表删除，如果释放，将其重新插入到链表的头部。如果内存块中的空闲节点不够用，则重新申请内存块，申请的内存块由内存块链表来管理。

  注意，本文涉及到的内存块链表和空闲内存节点链表的插入，为了省去遍历链表查找尾节点，便于操作，新节点的插入均是插入到链表的头部，而非尾部。当然也可以插入到尾部，读者可自行实现。

- ***内存池优点及缺点是什么？***

  - 比malloc/free进行内存申请、释放的方式要快
    - 因为每一次利用malloc申请内存，返回的都是void*指针，需要进行类型转换，速度要快一点
  - heap碎片产生较少
    - 因为是一次性开辟一定数量的连续内存空位作为内存池块，所以内存是集中在连续区域的，内存碎片相对好控制。且程序访问性能更好。
  - 对于频繁申请释放内存的情况能够提升性能
    - 因为内存是优先从已经申请过的内存池申请的，不够了，才再申请。如果是频繁申请，频繁释放的，很多时候都可以省去申请，直接分配。

#### 2.3 垃圾回收机制(GC)

- ***如何判断什么是垃圾以及垃圾回收原理***

  - **需要回收的垃圾都是在heap上的**

    ​	stack上的内存都会随着出栈的时候释放掉，而只有heap的数据一般是new出来的，由程序员自己管理。所以垃圾回收是对Heap的已经没有的资源进行回收。

  - ***什么时候一个资源才被认定为没有？***

    ​	`真正的死亡是世界上再没有一个人记得你--寻梦环游记`当一个资源已经跟大部分资源都没有关系的时候，就认为可以被释放了。而怎么选择一个合适的方法划分大部分呢？一般来说，较好的方法就是取stack上的数据。

    ​	如果一个资源跟stack上的所有数据都无关的时候，就认为是垃圾资源了。

  - ***总结***

    ​	GC将heap中每个已经分配的块看成一个节点，而所有已分配的块就构成了一个图。这个图里面，如果一个节点有对其他节点的引用，那么我们就称这两个节点是连通的。GC的原理就是判断根节点与哪些结点连通(连通分量)，而那些没有与根节点连通的节点就是无法访问到的垃圾，可以进行free。

    ​	所谓根节点，是指不在堆中分配的数据，如栈上的函数、数据区的全局变量或者寄存器等。可以想象与根节点连通就表示节点是”可达”的，我们可以使用或正在使用。而那些不可达的就应该回收。

    ​	简单来说GC机制，就是两步走标记联通块，不在联通块的就删除。

- ***GC什么时候运行？***

  ​	很显然地，跟家里什么时候搞卫生一样，要么勤快地如老妈子，定期搞；要么跟我一样，受不了了再搞卫生。

  - GC单独一个线程，周期性地回收
  - GC仅在申请空间的时候，检测一下free list，如果无法满足要求，就在出发垃圾回收机制

- ***GC具体实现算法***

  ​	GC具体实现算法以Mark & Sweep为代表，首先是要实现标记联通块目的。

  ```c++
  // isPtr()判断是不是会用到一个已经分配的块
  void mark (ptr p) {
  	if ((b = isPtr(p)) == NULL)
  		return;		// 如果是新开辟的则返回
  	if (blockMarked(b))
  		return;		// 如果已经标记则返回
  	markBlock(b);	// 标记这个块
  	len = length(b);
  	for (i=0; i < len; i++)
  		mark(b[i]);	//这个块的所有数据都进行标记
  	return;
  }
  
  // 从根节点出发，遍历所有块；将没有标记的块进行释放完成垃圾回收
  void sweep(ptr b, ptr end) {
  	while (b < end) {
  		if (blockMarked(b))
  			unmarkBlock(b);
  		else if (blockAllocated(b))
  			free(b);
  		b = nextBlock(b);
  	}
  	return;
  }
  ```
